VPN Fundamentals

🌐 What Is a VPN?

A Virtual Private Network (VPN) extends a private network across a public network (usually the internet), creating a logical, secure channel between two endpoints — so devices communicate as if on a private link.

Virtual → No physical link Private → Data protected Network → Multiple devices unified
🔴 THE PROBLEM: Public networks are insecure. Data without protection is vulnerable to eavesdropping, man-in-the-middle attacks, and surveillance — especially on open Wi-Fi.

🟢 THE SOLUTION: VPN creates an encrypted tunnel → your data becomes unreadable to any interceptor.

How a VPN Works — The Flow

The 4 Core VPN Functions

🔐 Encryption

Transforms readable data → ciphertext. Unauthorized parties intercepting traffic see only noise.

✅ Integrity

Detects if data was altered/tampered in transit. Ensures receiver gets exactly what was sent.

🪪 Authentication

Only authorized users/devices can establish a VPN. Prevents impersonation attacks.

🕵️ IP Masking

Hides real IP behind VPN server's address. Improves privacy & enables location flexibility.

Remember the 4 functions as EIAM → Encryption, Integrity, Authentication, Masking. Or think: "Every Intelligent Admin Masks" their traffic! 🛡️

Types of VPNs

📱 Remote Access VPN

Who: Individual users
What: Connects remotely to org's private network from anywhere
How: Authenticates each user, encrypts all traffic
Examples: Cisco AnyConnect, OpenVPN, WireGuard
Used for: Remote & hybrid work

🏢 Site-to-Site VPN

Who: Entire networks (not individuals)
What: Connects branch offices to HQ
How: Gateways at each site handle encryption transparently
Key: End-user devices need NO VPN config
Used for: Multi-branch enterprises

💻 Client-to-Site VPN

Who: Individual users (managed corp environment)
What: User installs VPN client software to connect into enterprise
How: IT enforces policies + access controls at gateway
Bonus: Often paired with MFA
vs Remote Access: More IT control

Remote Access = one person → organization (no IT management needed)
Site-to-Site = whole office → whole office (gateways do the work, users don't notice)
Client-to-Site = one person → organization (but IT controls everything via software + MFA)

IPsec Architecture

🔒 What Is IPsec?

IPsec (Internet Protocol Security) is a standardized suite of protocols that secures IP communications by authenticating and encrypting each IP packet in a data stream.

Layer 3 — Network Layer (OSI) Open standards — not proprietary Transparent to apps — no changes needed Packet-level protection
⚡ KEY DIFFERENCE from TLS: TLS works at the Application Layer (Layer 7). IPsec works at the Network Layer (Layer 3) — meaning it protects ALL traffic automatically, regardless of application. No app changes needed!

The 5 IPsec Security Services

🔐 Confidentiality

Encrypts packet contents using symmetric algorithms (AES). Intercepted data is unreadable without the key.

✅ Data Integrity

Uses cryptographic hash functions (SHA-256) + MACs to verify data wasn't altered. If hash doesn't match → packet discarded.

🔁 Anti-Replay

Each packet has a unique sequence number. Receiver tracks numbers and rejects duplicates → blocks replay attacks.

🪪 Origin Authentication

Verifies identity using digital signatures or pre-shared keys. Prevents impersonation & man-in-the-middle.

🔑 Key Management (IKE)

IKE automates negotiation, generation, and renewal of cryptographic keys. No manual distribution needed.

Remember the 5 services as C-I-A-R-K: Confidentiality, Integrity, Authentication, Replay-protection, Key management.
Or: "Can I Always Rely on Keys?" ✅

🤝 Diffie–Hellman Key Exchange

DH enables two parties to independently compute a shared secret over an untrusted channel — without ever transmitting the secret itself. The shared secret derives the symmetric encryption keys for the session.


• Neither party sends the actual key over the network
• An eavesdropper who captures the exchange CANNOT derive the secret
• DH Groups (e.g., Group 14, 19, 20) define key size & algorithm
• Foundational to IKE's secure key establishment

IPsec Protocols & Modes

🔵 AH — Authentication Header

PROVIDES:

✅ Data origin authentication
✅ Connectionless integrity (detects packet modification)
✅ Anti-replay protection (sequence numbers)
✅ IP header protection (authenticates outer header)


DOES NOT PROVIDE:

❌ Encryption — traffic is READABLE by interceptors (though not modifiable)

Use: Integrity only (no confidentiality needed)

🟠 ESP — Encapsulating Security Payload

PROVIDES:

✅ Confidentiality (encrypts payload with AES)
✅ Authentication (optional — verifies origin + integrity via MACs)
✅ Anti-replay (sequence number tracking)


MOST COMMONLY DEPLOYED protocol in modern VPN implementations due to encryption capability.

Use: Standard VPN — all modern deployments

AH vs ESP — Full Comparison

Feature AH (Auth Header) ESP (Encapsulating SP)
Encryption ❌ Not provided ✅ Yes (AES, 3DES…)
Data Integrity ✅ Yes ✅ Yes (optional)
Origin Authentication ✅ Yes ✅ Yes (optional)
IP Header Protection ✅ Includes outer header ⚠️ Partial (inner in tunnel mode)
Anti-Replay ✅ Yes ✅ Yes
Real-World Usage Rare — integrity-only scenarios ✅ Standard in ALL modern VPNs
💡 AH + ESP together? Can be used simultaneously when you need both full IP header integrity AND payload encryption — but this is uncommon in practice.

Operating Modes

🚀 Transport Mode

What's protected: Only upper-layer payload (TCP/UDP data)
IP Header: Remains intact & VISIBLE to routers
Use case: Host-to-host (peer-to-peer) communication
Overhead: Low (no extra IP header)
Weakness: Source/destination IPs visible → reveals network topology


Best for: End-to-end protection within trusted networks

🛡️ Tunnel Mode

What's protected: Entire original IP packet (header + payload)
IP Header: NEW outer header added (VPN gateway addresses)
Use case: Site-to-site VPN gateways
Overhead: Higher (new header added)
Strength: Internal topology completely hidden


Standard for enterprise VPN → preferred mode

Transport = "transport" your payload only → IP header visible → partial protection

🎯 Exam rule: Tunnel Mode = preferred for VPNs. Always.

IKE — Key Exchange

🔑 What Is IKE?

Internet Key Exchange (IKE) is the protocol that automates the negotiation, generation, and periodic renewal of cryptographic keys — eliminating the need for manual key distribution. IKE supports forward secrecy and handles peer authentication + SA (Security Association) establishment.

IKE — Two Phases

Phase 1 — Establish Secure Channel

Goal: Create a secure, authenticated channel (ISAKMP SA) between peers so Phase 2 can safely happen.

Steps in Phase 1:
1. Algorithm negotiation (encryption, hash, DH group)
2. Diffie-Hellman key exchange → shared secret
3. Peer authentication (PSK or digital certificates)
4. ISAKMP SA created

Main Mode Aggressive Mode

Phase 2 — Negotiate IPsec SA

Goal: Using the Phase 1 secure channel, negotiate the actual IPsec SAs used to protect data traffic.

Steps in Phase 2:
1. Negotiate IPsec policy (AH or ESP, algorithms)
2. Generate IPsec session keys
3. Establish IPsec SA (unidirectional)
4. Data can now flow securely

Quick Mode

Main Mode vs Aggressive Mode (Phase 1)

Feature Main Mode Aggressive Mode
Number of messages 6 messages 3 messages
Speed Slower ✅ Faster
Security ✅ More secure (identity protected) ⚠️ Less secure (identity exposed)
Identity protection ✅ Identity encrypted ❌ Identity sent in clear
Use case Default — most deployments When speed is critical

Main Mode = 6 messages, SLOW but SAFE (identity hidden) → "Main = Maximum security"
Aggressive Mode = 3 messages, FAST but RISKY (identity exposed) → "Aggressive = Attacks possible"

IKE Phase 1 = Build the secure tunnel
IKE Phase 2 = Build the data SA inside that tunnel

📝 Security Association (SA): A one-way logical connection between two devices that defines the security parameters (algorithm, keys, lifetime). Since it's one-way, a full bidirectional IPsec connection requires TWO SAs (one each direction).

Applications & Limitations

✅ IPsec Applications

🏢 Branch Office Connectivity

Site-to-site IPsec tunnels connect branches to HQ — seamless resource sharing without expensive dedicated leased lines.

🏠 Secure Remote Access

IPsec VPNs give remote employees encrypted access to internal systems — replacing insecure plain SSH or unprotected RDP.

🤝 Extranet & Intranet

Extranet: Controlled secure comms with trusted external partners.
Intranet: Between internal business units on segmented networks.

🛒 E-Commerce Security

Additional layer for payment systems & B2B transactions — supplements TLS at the network layer.

❌ IPsec Limitations

⚡ Processor Overhead

Crypto operations (encryption, decryption, integrity checking) impose significant CPU load. In high-throughput environments → reduced bandwidth + increased latency.

Solution: Hardware acceleration (dedicated crypto chips)

🧩 Network Design Complexity

Requires careful planning of Security Associations, key management, firewall rules, and NAT traversal.

Key ports/protocols to remember:

UDP 500 — IKE Protocol 50 — ESP UDP 4500 — NAT-T
⚠️ NAT Problem: IPsec and NAT don't play well together. NAT modifies IP headers, breaking AH integrity checks. Solution: NAT-T (NAT Traversal) — encapsulates IPsec in UDP port 4500.
⚡

Quick Reference Cheatsheet

🔢 NUMBERS TO MEMORIZE:
• DH key exchange = shared secret never transmitted
• IKE Phase 1 Main Mode = 6 messages
• IKE Phase 1 Aggressive Mode = 3 messages
• IKE = Phase 1 (secure channel) + Phase 2 (IPsec SA)
• UDP 500 = IKE | Protocol 50 = ESP | UDP 4500 = NAT-T
• SA is unidirectional → need 2 SAs for bidirectional
🎯 EXAM CRITICAL FACTS:
• IPsec = Layer 3 (Network) | TLS = Layer 7 (Application)
• AH = NO encryption | ESP = YES encryption
• Tunnel Mode = preferred for VPNs (hides topology)
• Transport Mode = host-to-host only
• AES = gold standard encryption algorithm
• HMAC-SHA-256 = current auth standard
• ESP = most deployed protocol
• Site-to-site = gateways handle crypto, not users
