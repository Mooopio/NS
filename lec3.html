Chapter 1: Foundations of Cryptography

Core definitions · Security goals · Key principles

🔐 What Is Cryptography?

The science of securing information by transforming it into an unreadable form — ensuring only authorized parties can access the original content.

Plaintext (readable data) → [Algorithm + Key] → Ciphertext (scrambled output)

Protects data during storage and transmission. Enables trust between parties who've never met. Without cryptography → no e-commerce, no banking, no private messaging.

⚠️ Why Cryptography? — The Threat Landscape

Every piece of data (email, password, payment) travels through untrusted networks. Attackers can:

INTERCEPT
Eavesdrop on messages and steal sensitive data in transit
MODIFY
Alter message contents without sender or receiver knowing
IMPERSONATE
Pretend to be a trusted party to deceive victims
Cryptography reduces risk — it is NOT "magic immunity." Proper implementation, key management, and protocol design are equally critical.

🎯 The 4 Security Goals — CIA + Non-Repudiation

CONFIDENTIALITY
Data remains secret & accessible only to authorized parties. Encryption converts readable data to ciphertext adversaries can't interpret.
INTEGRITY
Guarantees data has NOT been altered in transit or storage. Hash functions and MACs detect even single-bit unauthorized changes.
AUTHENTICATION
Verifies the IDENTITY of communicating parties. Digital certificates & signatures prove a message came from the claimed sender.
NON-REPUDIATION
Prevents a sender from DENYING they sent a message. Digital signatures create irrefutable proof tied to the signer's private key.

Remember: C.I.A.N. — Confidentiality, Integrity, Authentication, Non-repudiation. Think "CIA operatives Never deny their missions."

📚 Core Terminology

ENCRYPTION Converting plaintext → ciphertext using an algorithm + key. Makes data unreadable to unauthorized parties.
DECRYPTION Reverse process — transforming ciphertext back into plaintext using the correct key. Only authorized key holders can do this.
CLEARTEXT Data stored/transmitted WITHOUT encryption. Fully exposed and vulnerable to interception — a critical security risk.
PLAINTEXT The original readable data BEFORE encryption (or AFTER decryption).
CIPHERTEXT The scrambled, unreadable output AFTER encryption.

🔑 Kerckhoffs' Principle — The Big Idea

"The security of a system should depend SOLELY on the secrecy of the KEY — NOT on hiding the algorithm."
🔓
ALGORITHM
Public & peer-reviewed (AES, RSA, SHA-256). Strength proven by public scrutiny.
🗝️
KEY
SECRET. Strength comes from secrecy & randomness. PROTECT THIS.
🛡️
SECURITY
Strong keys + proven algorithms = security. Protect keys, not algorithms.

🎭 Steganography vs. Cryptography

🔐 Cryptography

  • Makes data UNREADABLE
  • Existence of message IS known
  • Content is NOT known
  • Interceptor sees ciphertext but can't understand it
  • Example: AES-encrypted file
VS

👻 Steganography

  • Makes data INVISIBLE
  • Hides data inside other media
  • Interceptor doesn't know message exists
  • Hides in images, audio, video pixel data
  • Example: text hidden in image pixels

Crypto = scramble the message. Stego = hide the message. They CAN be combined for layered security.

Chapter 2: Symmetric Encryption

Shared key · Block vs stream ciphers · Key distribution problem

🔁 Symmetric Encryption: The Concept

Same Key → Encryption & Decryption. Both sender and receiver MUST possess the shared secret BEFORE communicating.
⚡ SPEED
Extremely fast & efficient. Ideal for encrypting large volumes of data — files, disk partitions, network traffic.
🧮 SIMPLICITY
Mathematical operations (substitution, permutation, XOR) are computationally lightweight vs asymmetric.
❓ CHALLENGE
How do both parties securely get the same key? Intercepted key = entire system compromised.

📋 Symmetric Algorithms Comparison

Algorithm Key Size Status Notes
AES 128, 192, or 256-bit ✅ STANDARD Modern gold standard. NIST 2001. Used in TLS, disk encryption. Most secure & efficient.
DES 56-bit ❌ BROKEN Broken by brute force in 1999. NEVER use for security. Legacy only.
3DES 168-bit effective ⚠ PHASING OUT Applies DES 3 times. Successor to DES. Being phased out.
Blowfish Up to 448-bit LEGACY Variable-length keys. Bruce Schneier. Unbroken but less common in modern deployments.
Twofish 128, 192, 256-bit LEGACY AES finalist. Unbroken. Bruce Schneier. Less common in modern systems.

Remember: AES = ✅ Go. DES/3DES = ❌ Old. Blowfish/Twofish = Both by Schneier, both not common now.

🚧 The Key Distribution Problem

1

Interception Risk

If an attacker intercepts the key during exchange, they can decrypt EVERY future message — neither party may know the key was compromised.

2

Scalability Problem

In a network of n users, you need n(n-1)/2 unique keys. For 1,000 users → nearly 500,000 keys to manage securely.

3

The Solution

This problem motivated the invention of asymmetric cryptography and hybrid encryption systems.

Keys needed for n users = n × (n-1) / 2

📦 Block Ciphers vs. Stream Ciphers

📦 Block Ciphers

  • Encrypt data in fixed-size blocks (e.g., 128-bit for AES)
  • Same algorithm applied to each block
  • Requires padding if data doesn't fill final block
  • Dominant in modern systems
  • Examples: AES, DES, Blowfish
VS

🌊 Stream Ciphers

  • Encrypt one bit/byte at a time continuously
  • Generates pseudorandom keystream XORed with plaintext
  • Ideal for real-time data like voice/video
  • No padding required
  • Examples: ChaCha20, RC4 (deprecated)
CRITICAL: Correct usage of Initialization Vectors (IVs) and nonces is essential for BOTH types. Reusing an IV or nonce can CATASTROPHICALLY break security.

🖥️ Symmetric Encryption — Real-World Use Cases

💾 FILE & DISK ENCRYPTION
BitLocker, FileVault, VeraCrypt use AES to encrypt entire drives. Protected even if physical device is stolen.
🔒 VPN TUNNELS
IPSec, WireGuard establish encrypted tunnels. All traffic between endpoints protected from eavesdropping on public networks.
🗄️ DATABASE ENCRYPTION
Credit cards, health records, personal info encrypted at rest using symmetric keys managed by app or cloud provider.

Chapter 3: Asymmetric Encryption

Public/Private key pairs · RSA · ECC · Diffie-Hellman · Hybrid encryption

🗝️ Asymmetric Encryption: The Concept

A revolutionary breakthrough that solved the key distribution problem. Instead of one shared key, each party generates a mathematically linked key pair:

🌐 PUBLIC KEY
Shared openly with the world. Anyone can use it to ENCRYPT messages intended for the key owner.
🔒 PRIVATE KEY
Kept STRICTLY SECRET. Only the owner can use it to DECRYPT messages or CREATE digital signatures.
Enables secure communication WITHOUT prior key exchange. Forms the basis of digital signatures, certificates, and HTTPS.

↔️ How Public & Private Keys Are Used — Dual Roles

🔐 CONFIDENTIALITY FLOW
Encrypt with recipient's PUBLIC key
Only recipient's PRIVATE key can decrypt
→ Ensures only intended party can read message
✍️ AUTHENTICITY FLOW
Sign with sender's PRIVATE key
Anyone with sender's PUBLIC key can verify
→ Proves message came from claimed sender & wasn't altered

Encrypt → use receiver's PUBLIC key. Sign → use sender's PRIVATE key. Think: "PUBLIC locks the box, PRIVATE signs the letter."

⚙️ Asymmetric Algorithms

AlgorithmBased OnKey SizeUseNotes
RSA Difficulty of factoring large primes 2048–4096 bits Encryption + Signatures Most widely deployed. Invented 1977. Foundation of TLS/SSL.
ECC Elliptic curve discrete logarithm 256-bit (≈3072-bit RSA) Encryption + Signatures Same security as RSA with MUCH smaller keys. Preferred for mobile & IoT.
Diffie-Hellman Discrete logarithm Varies Key Exchange ONLY NOT encryption. Creates shared secret over insecure channel. Basis for Perfect Forward Secrecy in TLS.

RSA = Rivet, Shamir, Adleman (inventors). ECC = tiny keys, same power. DH = "I don't encrypt, I just agree on a secret."

🎨 Diffie-Hellman Key Exchange — Paint Mixing Analogy

1

Agree on Common Color (Public)

Alice and Bob both pick yellow paint publicly. Everyone can see this.

2

Mix with Secret Color (Private)

Alice secretly adds red. Bob secretly adds blue. Private colors are NEVER shared.

3

Exchange Mixed Colors (Public Channel)

They send their mixed colors over the public channel. Eavesdropper sees the mixtures but cannot reverse the mixing.

4

Derive Shared Secret

Each adds their private color to the received mix → same final color. This shared secret becomes the symmetric session key.

The shared secret is then used as a SYMMETRIC session key for fast bulk encryption.

🔀 Hybrid Encryption — Why It Exists & How It Works

Neither symmetric nor asymmetric encryption is perfect alone. Hybrid = combines their strengths. This is how ALL modern secure protocols work.

1

Asymmetric Handshake

Use RSA or Diffie-Hellman to securely exchange a random symmetric session key. Asymmetric crypto solves key distribution.

2

Symmetric Data Transfer

Use the exchanged session key with AES-GCM for fast bulk encryption. Symmetric crypto provides the speed needed for real-time data.

Result: Secure key exchange + high-speed encryption. This is exactly how TLS/HTTPS works every time you visit a secure website.

Chapter 4: Hash Functions & Integrity

One-way transforms · Password hashing · MACs · HMACs

🔩 Hash Functions: The Concept

A one-way mathematical transformation that takes an input of ANY size and produces a fixed-length output called a hash (or digest). Unlike encryption, hashing is NOT reversible.

📏
ANY INPUT SIZE
1 character, a novel, or 4GB file → all produce same fixed-length hash
📌
FIXED OUTPUT
SHA-256 ALWAYS produces 256 bits regardless of input length
🌊
AVALANCHE EFFECT
Changing even ONE bit of input produces a completely different hash output

⚖️ Hash Algorithms — Know Your Status

AlgorithmOutput SizeStatusUse Case
MD5 128-bit ❌ BROKEN Collision attacks demonstrated 2004. NEVER use for security. Legacy checksum only.
SHA-256 256-bit ✅ RECOMMENDED Current industry standard. Digital signatures, certificates, integrity verification. No practical attacks known.
bcrypt 192-bit ✅ PASSWORDS Designed specifically for PASSWORD hashing. Intentionally SLOW by design. Configurable work factor.
scrypt / Argon2 Variable ✅ PASSWORDS Memory-hard. Even better than bcrypt for passwords. Resistant to GPU/ASIC attacks.

SHA-256 = for data integrity. bcrypt/scrypt/Argon2 = for passwords (SLOW is GOOD here). MD5 = DEAD. Never use MD5 for security.

🔑 Password Hashing: How To Do It RIGHT

1

Never Store Plaintext

If attacker gains DB access, plaintext passwords are IMMEDIATELY compromised. Always store only the hash.

2

Use a Unique Salt Per User

A salt is a random value prepended to each password before hashing. Prevents rainbow table attacks. Ensures identical passwords produce different hashes.

3

Use Slow, Strong Schemes

Standard SHA-256 is TOO FAST for passwords — attackers try billions per second. Use bcrypt, scrypt, or Argon2 — deliberately slow and memory-hard.

SHA-256 is GREAT for file integrity but TERRIBLE for passwords. The speed that makes it good for files makes it dangerous for passwords.

🔏 Plain Hash vs. MAC vs. HMAC

#️⃣ Plain Hash

  • Anyone can compute the hash of any data
  • Detects accidental corruption
  • Does NOT authenticate the sender
  • No secret key involved
  • Attacker can modify data & recompute hash
USE CASE: File download integrity checks
VS

🔐 MAC / HMAC

  • Combines hash with a shared secret key
  • Only parties with the key can create or verify
  • Authenticates both integrity AND origin
  • HMAC = Hash-based MAC (e.g., HMAC-SHA256)
  • Attacker without key CANNOT forge
USE CASE: API authentication, message integrity in protocols

Plain Hash = knows data changed. MAC/HMAC = knows data changed AND who sent it. Always prefer HMAC when authentication matters.

Chapter 5: Digital Signatures

Sign · Verify · Authenticate · Non-repudiation

✍️ Digital Signatures: What & Why

The cryptographic equivalent of a handwritten signature — but far more secure. Provides mathematical proof that a message/document was created by a specific party and hasn't been altered since signing.

✅ AUTHENTICITY
Proves message was created by claimed sender, verified via their PUBLIC key.
🔒 INTEGRITY
ANY modification to the signed data invalidates the signature — tampering is immediately detectable.
🚫 NON-REPUDIATION
Signer CANNOT deny having signed. Only their private key could have produced the signature.

⚙️ How Digital Signatures Work — Step by Step

Two distinct phases: SIGNING (sender side) and VERIFYING (receiver side).

SIGNING PHASE (SENDER)
1

Hash the Message

Run the document through SHA-256 → produces digest.

2

Sign the Hash

Encrypt the hash with sender's PRIVATE key → this IS the signature.

VERIFYING PHASE (RECEIVER)
3

Verify Signature

Decrypt the signature using sender's PUBLIC key → recovers the original hash.

4

Confirm Match

Hash received document independently. Compare both hashes — match = valid, mismatch = tampered/forged.

This process runs BILLIONS of times daily in TLS certificates, software updates, and document signing.

Sign = Hash → Encrypt with PRIVATE key. Verify = Decrypt with PUBLIC key → Compare hashes. Two hashes match = ✅ authentic.

Chapter 6: Real-World Applications

Where cryptography lives in everyday technology

🌍 Daily Cryptographic Touchpoints

1

🔓 Unlock Phone

Biometric data verified against encrypted template. Full-disk encryption protects stored data.

2

🌐 Browse HTTPS

TLS handshake: certificate verification → Diffie-Hellman key exchange → AES-GCM session encryption. All automatic.

3

💬 Send a Message

End-to-end encryption (Signal Protocol: DH + AES + HMAC). Ensures only sender and recipient can read the conversation — not even the service provider.

4

💳 Make a Payment

Card data encrypted in transit, tokenized at the processor, signed for non-repudiation.

💬 Secure Communications In Practice

📱 E2EE MESSAGING
Signal, WhatsApp, iMessage. Uses Signal Protocol: Diffie-Hellman + AES + HMAC. Even the service provider CANNOT read messages.
📧 EMAIL ENCRYPTION
PGP (Pretty Good Privacy) & S/MIME use hybrid encryption. Sender encrypts with recipient's public key. Signatures verify authenticity.
📞 VOIP & VIDEO
SRTP (Secure Real-Time Protocol) with symmetric keys via DTLS. Real-time encryption without perceptible latency.

🎯 Key Takeaways — What to Remember Forever

🔑
Protect Keys, Not Algorithms
Security depends on key secrecy and proper management — not hiding the algorithm.
🔀
Hybrid Is the Standard
Real-world systems combine asymmetric key exchange with symmetric bulk encryption.
✅
Integrity Matters Too
Use AEAD modes and digital signatures — encryption alone isn't enough.
⏩
Stay Current
Algorithms age. Migrate away from broken primitives: MD5, DES, RC4, 3DES. Prepare for post-quantum era.
